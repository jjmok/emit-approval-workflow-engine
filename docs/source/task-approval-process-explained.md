# **Task Approval System**

*A simple explanation of how the whole process works*

This system manages tasks that need approval — things like invoice approvals or staff claims. Each task travels through the right people in the right order, with clear deadlines, automatic reminders, and a complete record of everything that happened. The goal is simple: nothing gets stuck, and nothing slips through the cracks.

# **1\. Where a task comes from**

Every task is built from a standard template, so the same kind of request always captures the same information. A task can start in one of two ways:

* On a schedule (recurring) — the system creates it automatically at set times. If a new one is due while an earlier one is still open, both simply run side by side.

* On demand (one-off) — someone raises it when needed.

# **2\. How a task moves**

A task moves through a series of steps, one after another. Each step belongs to one team, and the task only moves to the next step once the current step is approved.

The number of steps is flexible — a task type can have one team or several (for example: Team A → Team B → a final approver). Teams can be added, removed, or reordered just by changing the setup, with no rebuild needed.

# **3\. Who handles each step**

Within a team, a step goes to a named person in charge (PIC) first. If they don’t act in time, it automatically passes to the next person, and finally to the team boss:

* First choice: PIC \#1

* If no response in time: PIC \#2

* Still no response: the team boss

Each person gets their own response window (counted in working hours). When it runs out, the task hands off to the next person with a fresh window — so a single slow person never holds things up for long.

# **4\. Two kinds of deadlines**

Two clocks run at the same time on every step:

**The personal clock:** how long each individual has to respond before it passes to the next person.

**The step clock (KPI):** the overall target time for the whole step. If the step takes too long, the task is pushed up the management chain — team boss, then boss of boss, then a single company-wide top person — one rung at a time for every extra half-day it stays overdue.

To keep this tidy, both clocks feed one simple rule: a task only ever moves up to a more senior person, never back down. If it’s already with the boss, a later deadline miss won’t send it back to a junior person.

# **5\. The three things a reviewer can do**

## **Approve**

The task moves on to the next step, or finishes if it was the last one.

## **Reject**

A reason is required. The rejection goes to the boss for a second look. If the boss agrees the work is fine, the task carries on from exactly where it was. If the boss also rejects, it goes back to the person who raised it to fix and resubmit.

## **Ask the creator for more information**

The reviewer can send a question back to the person who raised the task, without rejecting it. While they wait for an answer, the step’s clocks pause — the reviewer isn’t penalised for someone else’s delay — and the creator gets their own deadline to reply. If the creator goes quiet, the question moves to the creator’s boss. The reviewer can also withdraw the question at any time and decide with what they have, so a task never gets stuck waiting forever.

# **6\. Covering for each other**

Anyone can forward a task to a colleague — either just once, or as a standing arrangement for a date range (for example, while on leave). The stand-in acts in the original person’s place, the overall step deadline stays exactly the same, and the task continues on its normal path afterwards. Cover assignments get a shorter half-day window to keep things moving.

# **7\. Staying informed**

People are notified on their mobile when a task lands with them, when a reminder is due, and when something is escalated. Mobile notifications are the main way the system keeps everyone moving.

# **8\. Keeping score**

A dashboard shows how many tasks each team has, whether steps are being completed on time, and where things are getting stuck.

* A late step counts as one breach against the team that owns it — blame sits with the team, not one individual.

* Each step counts only once, no matter how many times it then escalates, so the numbers never get inflated.

* Bosses see their own team; more senior leaders see the teams beneath them.

Behind the dashboard, every single action is recorded in an audit trail that can never be changed — who did what, when, and why — giving a full, trustworthy history of each task.

# **Timing at a glance**

| What | Clock | If missed |
| :---- | :---- | :---- |
| A person responding to a step | Their own response time (about a day) | Passes to the next person, then the team boss |
| Whole step finishing in time | The step KPI (the target time for that step) | Escalates up: team boss → boss of boss → company top person |
| Someone covering during leave | A shorter half-day window | Passes back into the step's normal chain |
| Creator answering a question | Their own response time | Passes to the creator's boss, then waits there |

*All deadlines are measured in working hours — the clocks pause on evenings, weekends, and public holidays, so nobody is penalised for time the office was closed.*
